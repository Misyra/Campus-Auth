**日期**: 2026-06-13  
**项目版本**: 4.0.2

---

## 1. 概述

基于 `docs/code-review-report.md` 的二次审查结果，修复所有验证属实的中低严重度问题。采用"按模块分组验证并修复"的策略，每个模块的问题一起处理，验证属实的问题立即修复并附带测试用例。

### 1.1 修复目标

- 修复所有验证属实的中低严重度问题
- 每个修复都附带对应的测试用例
- 记录验证结果（属实/不属实/已修复/已不适用）

### 1.2 不在范围内

- 高严重度问题（已修复或不属实）
- 测试覆盖率提升（单独规划）
- 测试 fixture 重构（单独规划）
- 代码风格统一（单独规划）

## 2. 修复流程

### 2.1 总体流程

```
对每个模块：
  1. 读取该模块的所有问题
  2. 逐个验证问题是否确实存在
  3. 如果存在：
     - 修复问题
     - 写测试用例
     - 记录修复结果
  4. 如果不存在：
     - 跳过并记录原因
  5. 提交该模块的所有修复
```

### 2.2 验证方法

对每个问题，验证方法如下：

1. **读取当前代码**：检查问题是否仍然存在
2. **检查修复历史**：查看 git log 是否已有相关修复
3. **运行测试**：确认问题是否可复现
4. **记录验证结果**：
   - `属实`：问题存在，需要修复
   - `不属实`：问题不存在，代码已正确
   - `已修复`：问题曾存在但已被修复
   - `已不适用`：问题描述的场景已不存在

### 2.3 测试策略

每个修复都附带测试用例：

- **正确性测试**：验证修复后行为正确
- **边界测试**：测试边界条件和异常情况
- **回归测试**：确保修复不会引入新问题

### 2.4 提交规范

每次提交格式：

```
fix: 修复 <模块名> 中的 <问题类型>

- 修复 <具体问题>
- 添加 <测试用例>
- 验证结果：<属实/不属实>
```

## 3. 分组方案

### 3.1 模块分组

| 模块 | 问题数量 | 关键问题 |
|------|---------|---------|
| **API 路由层** (app/api/) | 6 个中 + 13 个低 | 内存耗尽、竞态条件、回滚语义、依赖注入 |
| **服务层** (app/services/) | 5 个中 + 5 个低 | 线程安全、封装违反、文件写入、危险删除 |
| **工具层** (app/utils/) | 7 个中 + 7 个低 | 进程返回值、超时配置、同步阻塞 |
| **前端** (frontend/) | 3 个中 + 9 个低 | CSS 注入、空引用、运算符优先级 |
| **入口文件** (main.py, start.go 等) | 6 个中 + 9 个低 | 进程终止、event loop、文件清理 |
| **测试** (tests/) | 3 个中 + 若干低 | 无效测试、fixture 重复 |

### 3.2 处理顺序

按模块依赖关系排序，先修复底层模块，再修复上层模块：

1. **工具层** (app/utils/) — 基础工具，被其他模块依赖
2. **服务层** (app/services/) — 业务逻辑层
3. **API 路由层** (app/api/) — 接口层
4. **入口文件** (main.py, start.go 等) — 启动入口
5. **前端** (frontend/) — 用户界面
6. **测试** (tests/) — 测试套件

## 7. 风险与缓解

### 7.1 风险

1. **验证不准确**：可能误判问题是否存在
2. **修复引入新问题**：修复可能引入新的 bug
3. **测试覆盖不足**：测试用例可能不够全面

### 7.2 缓解措施

1. **多次验证**：对不确定的问题，多次验证
2. **运行全量测试**：每次修复后运行全量测试
3. **代码审查**：修复后进行代码审查
