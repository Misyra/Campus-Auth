# Not To Do

> 经审查判定为不需要做的事项。这些是功能/策略层面的设计决策，不是 bug。不要尝试"修复"它们。

---

## 安全类 — 本地单用户桌面应用，无需服务端级安全加固

- **不要加服务端安全加固** — 应用仅监听 127.0.0.1，无远程访问面。不要添加 API 鉴权、CORS 限制、SSRF 防护、输入消毒、沙箱、路径遍历防护（TASK_ID_PATTERN 已排除穿越字符）、参数黑白名单等。用户已拥有完整系统权限，这些措施不适用。
- **不要脱敏日志和错误信息** — 本地单用户，详细日志和错误信息帮助排查问题。不要对 repo_proxy、browsers、启动日志、脚本执行等模块的日志/异常做脱敏或掩码处理。
- **不要加密码额外保护** — 当前 Fernet 加密 + 密钥与配置分离存储已是合理方案。不要再加 API 密码掩码、密钥内存擦除（Python 字符串不可变）、密码注入防护等。
- **不要给脚本执行加沙箱或额外限制** — 用户自己编写和配置的脚本，用户已拥有完整系统权限，增加沙箱/路径规范化/参数过滤无收益。
- **不要阻止用户自配的 URL 请求** — 用户配置的 portal URL、repo_proxy URL 等，verify=False 是兼容自签名证书的设计，不要加 SSRF 防护或协议以外的限制。

## 并发/竞态类 — 单用户不会触发并发冲突

- **不要加细粒度并发保护** — 单用户桌面应用不会触发并发竞态。CPython GIL 保证基本原子性，不需要对 bool 标志位、配置保存、stop() 流程等加锁或竞态保护。
- **不要给单用户操作加并发限制** — 前端无批量执行功能，不可能并发大量请求，不需要给端点加并发控制。

## 架构类 — 核心设计决策，不要重构

- **不要缩窄 `except Exception`** — 约 130 处防御性捕获全部配有日志，桌面应用"永不崩溃"是核心目标。Playwright/httpx/网络操作的异常类型不可穷举，缩窄只会引入漏网崩溃。
- **不要将 threading 架构迁移为 asyncio** — engine/worker/tray 等核心组件基于 daemon thread + 队列通信，FastAPI 仅用于 API 层。统一为 asyncio 牵动整条链路，收益不抵风险。
- **不要给前端加 TypeScript 或构建工具链** — 原生 HTML/JS/CSS 无构建步骤，用户可直接修改前端文件；加入 bundler 会破坏这个优势。
- **不要拆分 engine.py 和 playwright_worker.py** — 它们分别是调度和浏览器自动化的核心，内部职责紧密，拆分只会增加模块间耦合。

## 功能行为类 — 故意这样设计的

- **网络探测用严格模式** — 当前"任一探测失败即返回 False"是故意设计：宁可误报断网触发多余登录，不可漏报导致断网不处理。不要改为"任一成功即返回 True"。
- **凌晨暂停期默认启用** — 校园网凌晨 0-6 点一般不需要认证，空配置默认启用暂停是故意设计。
- **WS 重连耗尽后等待用户交互触发** — 耗尽后由 visibilitychange 在用户切回页面时主动重连，后台低频探活会在后端长期故障时产生无效请求与日志噪声。
- **无 monitor_config 时登录成功判定为 True** — 无网络检测配置时只校验任务步骤是否执行成功，改为 False 会导致无配置场景下登录永远失败。

## 代码质量类 — 功能正确，不要过度优化

- **不要给任务保存加 Pydantic 严格校验** — 任务 JSON 结构需要灵活性，严格校验可能拒绝合法任务。
- **不要过度优化前端** — 不需要给 API 调用加 loading 状态（单用户并发概率低）、改拖拽状态为 Vue 响应式（单实例模块级变量正常工作）、优化 configDirty 的 JSON.stringify（配置对象体量下不会卡顿）。
- **不要提取前端小工具函数** — 如 applyAppearance 颜色解析、quitApp innerHTML 等，功能正确且使用场景有限，提取公共函数属于不必要的抽象。
- **不要擦除密钥内存** — Python 字符串不可变，bytearray 方案需重构整个接口，收益不抵成本。

## 兼容性/边缘场景 — 不处理

- **不处理非中文 Windows 的网关编码差异** — 校园网场景主要覆盖中文/英文系统。
- **不处理十六进制 SSID 误判** — 实际校园网 SSID 不会是纯十六进制字符串，概率极低。

## 误判说明 — 以下模式是正确的，不要标记为问题

- 代码审查工具可能对以下模式报出警告，这些都是正确的，不需要修复：
  - asyncio 原语均在事件循环线程中使用（`system.py` 的 `shutdown_event.set()` 从线程池调用是唯一例外，但仅在关闭时触发一次，实际无影响，不要加 `call_soon_threadsafe`）
  - `os._exit()` 的正常路径（`force_exit`）已做 atexit 清理；紧急路径（双击 Ctrl+C、tray fallback）故意跳过清理以实现快速退出
  - `cryptography` 是必需依赖，缺失时回退明文存储并 warning
  - CLOSE-then-RELEASE 是正确的浏览器资源管理顺序
  - DOM detached 节点由 GC 正常回收；beforeUnmount 已清除所有 timer 和事件监听器
