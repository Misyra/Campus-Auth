#### F01 — 配置回滚检查第二次 reload 返回值

**文件**: `app/services/config_service.py:112-128`

**当前代码**（第 119 行）：
```python
reload_fn()                    # 返回值被丢弃
```

**问题**: `reload_fn()`（即 `engine.reload_config()`）返回 `tuple[bool, str]`，正常不抛异常。第二次（回滚后）重载失败时返回 `(False, msg)`，但 `try/except` 不捕获（无异常），返回值被丢弃。磁盘已回滚，运行时却停留在第一次失败的状态，且最终返回给用户的 `message` 是第一次的 msg。

**修复**:
```python
def save_and_apply(payload, profile_service, reload_fn) -> SaveResult:
    backup_data = copy.deepcopy(profile_service.load())
    save_config_combined(payload, profile_service)

    ok, msg = reload_fn()
    if ok:
        return SaveResult(success=True, message="配置保存成功")

    # 第一次重载失败，尝试回滚
    config_logger.error("配置重载失败，正在回滚: {}", msg)
    try:
        profile_service.update(lambda data: _rollback_config(data, backup_data))
        rollback_ok, rollback_msg = reload_fn()          # CHANGED: 捕获返回值
        if not rollback_ok:
            # 回滚重载也失败：两次都失败，磁盘已回滚但运行时可能不一致
            config_logger.error(
                "配置回滚后重载仍失败: {}（磁盘已回滚，运行时状态可能不一致）",
                rollback_msg,
            )
            return SaveResult(
                success=False,
                message=f"配置重载失败: {msg}（回滚后仍失败: {rollback_msg}）",
            )
        config_logger.warning("配置已回滚并重载成功（原失败: {}）", msg)
        return SaveResult(success=False, message=f"配置重载失败，已回滚: {msg}")
    except Exception as rollback_exc:
        config_logger.error(
            "回滚过程异常（磁盘配置已回滚，运行时状态可能不一致）: {}",
            rollback_exc,
            exc_info=True,
        )
        return SaveResult(success=False, message=f"配置重载失败且回滚异常: {msg}")
```

**测试要点**: mock `reload_fn` 第一次返回 `(False, "超时")`、第二次返回 `(False, "又超时")`，断言 SaveResult.message 同时包含两次失败信息。

#### F02 — login_once 记录登录历史

**文件**: `main.py:195-247`

**问题**: `_execute_login_with_retries` 直接 `get_worker().submit(CMD_LOGIN, ...)`，完全绕过 `TaskExecutor`，其 `_record_login_history()`（task_executor.py:335/340/346/351）不会被触发。`--startup-action login_once` 的登录在历史页面不可见。

**修复思路**: 该函数运行在进程启动早期，尚未构造 `ServiceContainer`（容器在 `_run_lightweight`/`_run_full` 才创建）。完整复用 TaskExecutor 需要重构启动顺序，成本高。**采用轻量方案**：直接构造 `LoginHistoryService` 记录历史。

```python
def _execute_login_with_retries(runtime_config: dict, logger) -> LoginResult:
    from app.constants import AUTH_DATA_DIR
    from app.services.login_history_service import LoginHistoryService
    from app.services.profile_service import create_profile_service
    from app.workers.playwright_worker import CMD_LOGIN, get_worker

    # NEW: 用于记录登录历史（与 TaskExecutor 内部一致的写法）
    profile_service = create_profile_service()
    history = LoginHistoryService(AUTH_DATA_DIR)

    retry_settings = runtime_config.get("retry_settings", {})
    raw = retry_settings.get("max_retries", 3)
    max_retries = max(1, min(raw, 10))
    retry_interval = int(retry_settings.get("retry_interval", 5))

    attempt = 0
    while True:
        attempt += 1
        if attempt > 1:
            delay = min(retry_interval * (2 ** (attempt - 2)), 300)
            print(f"等待 {delay} 秒后重试第 {attempt} 次...")
            time.sleep(delay)

        start = time.perf_counter()
        success = False
        message = ""
        try:
            result = get_worker().submit(
                CMD_LOGIN,
                data={"config": runtime_config},
                timeout=120,
            )
            success = result.success
            message = result.data if result.success else result.error or "登录失败"
        except Exception as exc:
            message = f"登录异常: {exc}"

        duration_ms = int((time.perf_counter() - start) * 1000)
        # NEW: 记录历史（成功/失败都记，与 TaskExecutor 行为一致）
        try:
            history.record(
                success=success,
                duration_ms=duration_ms,
                profile_service=profile_service,
                error="" if success else message,
            )
        except Exception:
            logger.debug("记录登录历史失败", exc_info=True)

        if success:
            print(f"登录成功: {message}")
            cleanup_orphan_browsers()
            return LoginResult.SUCCESS

        print(f"登录失败 (第 {attempt} 次): {message}")
        if attempt >= max_retries:
            break

    cleanup_orphan_browsers()
    print(f"已重试 {max_retries} 次均失败，回退到正常模式")
    logger.warning("登录失败（已重试 {} 次），回退到正常模式", max_retries)
    return LoginResult.TEMPORARY_FAILURE
```

**注意**: `LoginHistoryService.record` 的签名需复核（task_executor.py:487 调用方式：`record(success=, duration_ms=, profile_service=, error=)`）。此方案与 TaskExecutor 内部写法完全对齐，历史记录格式统一。

**备注**: F08 会进一步统一 login_once 的重试机制。本批先保证历史可见，避免一次性改动过大。

#### F03 — record_attempt 移到 execute_login_async 成功提交之后

**文件**: `app/services/engine.py:326-335`

**当前代码**:
```python
if not is_manual:
    self._login_retry.record_attempt(time.time())   # count+1

try:
    future = self._task_executor.execute_login_async(config_snapshot=config_snapshot)
```

**问题**: 若 `execute_login_async` 抛异常（如线程池已关闭），count 已递增但登录未执行，白白消耗一次重试机会，最终可能因 count 达到 max_retries 而停止重试。

**修复**（对调顺序）:
```python
try:

except Exception:
    self._update_status_snapshot()
    raise
if not is_manual:                                    # CHANGED: 移到提交成功之后
    self._login_retry.record_attempt(time.time())
```

**注意**: `execute_login_async` 在命中去重时返回旧 future（task_executor.py:197），此时旧 future 已经是上一次成功提交的登录，count 在上次提交时已 +1，此处不再重复 +1，行为正确。

#### F04 — 网络检测不再无条件 reset 重试计数

**文件**: `app/services/engine.py:258-291`

**当前代码**:
```python
if result.get("need_login", False):
    self._login_retry.reset()          # ← 每次 need_login 都清零
    self._configure_retry()
    self._do_async_login()
else:
    self._login_retry.reset()          # ← 网络恢复也清零
```

**问题**: 网络检测每 5 分钟（或配置间隔）执行一次。need_login=True 时 reset 使 count 归零，下一轮网络检测又触发新一轮 max_retries 重试。认证服务器长期宕机时，系统永不停机地循环"检测→重试系列→检测→重试系列"，每轮都启动浏览器。

**修复策略**: 引入"连续失败"计数，超过阈值后进入"降频检查"模式，只在状态变化时才快速重试。

```python
# __init__ 中新增字段
self._consecutive_login_failures: int = 0
self._backoff_check_multiplier: int = 1   # 降频倍数

def _do_network_check(self) -> None:
    core = self._monitor_core
    if core is None:
        return

    try:
        result = core.check_once()
        interval = int(result.get("interval", self._monitor_check_interval))

        if result.get("need_login", False):
            # CHANGED: 仅在首次发现 need_login（重试尚未进行）时 reset+configure
            if self._login_retry.count == 0:
                self._login_retry.reset()
                self._configure_retry()
            self._do_async_login()
        else:
            # 网络恢复正常：清空失败计数，重置重试状态
            if self._consecutive_login_failures > 0:
                logger.info("网络已恢复，重置重试状态")
            self._consecutive_login_failures = 0
            self._backoff_check_multiplier = 1
            self._login_retry.reset()

        if core.consume_profile_switch_flag():
            logger.info("检测到方案切换，准备重启监控")
            if self._reload_config_internal():
                self._handle_stop()
                self._handle_start(EngineCommand(type=EngineCmdType.START))
            else:
                logger.error("配置重载失败，继续使用当前配置")

        self._next_network_check = time.time() + interval
        self._update_status_snapshot(force=True)
    except Exception:
        logger.exception("网络检测异常")
        self._next_network_check = time.time() + self._monitor_check_interval
```

**配套：在 `_do_async_login` 的 `_on_done` 回调中累计失败次数并实现降频**：
```python
def _on_done(f: Future) -> None:
    self._update_status_snapshot()
    try:
        ok, msg = f.result()
        tag = "手动登录" if is_manual else "自动登录"
        if ok:
            logger.info("{}完成: {}", tag, msg)
            # NEW: 成功则清空失败计数
            if not is_manual:
                self._consecutive_login_failures = 0
        else:
            logger.warning("{}失败: {}", tag, msg)
            # NEW: 累计失败，达到阈值后降频
            if not is_manual:
                self._consecutive_login_failures += 1
                if self._consecutive_login_failures >= self._login_retry_max_cycles():
                    self._apply_backoff_interval()
    except Exception:
        logger.exception("登录任务异常")

# 新增辅助方法
_LOGIN_BACKOFF_THRESHOLD = 3   # 连续 3 轮（每轮 max_retries 次）失败后降频

def _login_retry_max_cycles(self) -> int:
    if self._login_retry.config:
        return self._LOGIN_BACKOFF_THRESHOLD
    return self._LOGIN_BACKOFF_THRESHOLD

def _apply_backoff_interval(self) -> None:
    """达到失败阈值后，延长下一次网络检测间隔（指数退避，上限 30 分钟）。"""
    self._backoff_check_multiplier = min(self._backoff_check_multiplier * 2, 6)
    extra = (self._backoff_check_multiplier - 1) * self._monitor_check_interval
    self._next_network_check = time.time() + extra
    logger.warning(
        "登录已连续失败 {} 轮，下次网络检测降频至 {}s 后",
        self._consecutive_login_failures,
        int(extra),
    )
```

**设计要点**: 重试管理器（`LoginRetryManager`）本身不变，仍按 `max_retries` 停止。降频发生在**网络检测层**：连续多轮全失败后，推迟下一次 `check_once`，避免认证服务器宕机时的无意义浏览器启动。网络一恢复（need_login=False），立即清零、恢复正常检测频率。

### 第二批：手动登录体验与启动顺序（F05–F07）

#### F05 — 自动登录路径增加配置校验

**文件**: `app/services/engine.py:313-354`（`_do_async_login`）

**问题**: `_handle_login`（手动入口，420-422）校验 username/password/auth_url；但 `_do_network_check → _do_async_login`（自动入口）无校验。配置不完整时，空配置传入 Worker，启动浏览器后才在步骤级失败，浪费 5-15 秒。

**修复**: 在 `_do_async_login` 顶部统一校验，手动和自动路径都覆盖。同时把 `_handle_login` 的校验也改为调用同一逻辑，消除重复。

```python
def _validate_login_config(self, config: dict | None = None) -> str | None:
    """校验登录配置完整性。返回 None 表示通过，否则返回错误信息。"""
    cfg = config if config is not None else self._copy_runtime_config()
    if not cfg.get("username") or not cfg.get("password") or not cfg.get("auth_url"):
        return "登录配置不完整（请先设置认证地址、用户名和密码）"
    return None

def _do_async_login(self, is_manual: bool = False, config_snapshot: dict | None = None) -> bool:
    # NEW: 统一配置校验（自动登录路径此前缺失）
    config = config_snapshot if config_snapshot is not None else self._copy_runtime_config()
    err = self._validate_login_config(config)
    if err is not None:
        logger.warning("跳过自动登录: {}", err)
        self.record_log(err, level="WARNING", source="backend")
        # 重置重试状态，避免持续触发
        self._login_retry.reset()
        return False

    if self._task_executor.is_login_running():
        if not is_manual:
            return False
        logger.info("手动登录：取消当前登录任务")
        self._task_executor.cancel_login()
        deadline = time.time() + 5
        while self._task_executor.is_login_running() and time.time() < deadline:
            time.sleep(0.1)
        if self._task_executor.is_login_running():
            logger.warning("取消当前登录超时，将尝试提交新登录")
    if not is_manual:
        self._login_retry.record_attempt(time.time())

    try:
        future = self._task_executor.execute_login_async(config_snapshot=config)
    except Exception:
        self._update_status_snapshot()
        raise
    # ... 后续 add_done_callback 不变 ...
```

**同步修改 `_handle_login`**（消除重复校验）:
```python
def _handle_login(self, cmd: EngineCommand) -> None:
    config = self._copy_runtime_config()
    err = self._validate_login_config(config)            # CHANGED: 复用
    if err is not None:
        cmd.response_data = (False, err)
        return
    if self._do_async_login(is_manual=True, config_snapshot=config):
        cmd.response_data = (True, "登录已提交")
    else:
        cmd.response_data = (False, "登录任务已在执行中，请稍后再试")
```

#### F06 — 修复手动取消竞态窗口

**文件**: `app/services/engine.py:329-332` + `app/services/task_executor.py:168-208`

**问题**: 手动取消旧登录超时后，`_do_async_login` 调 `execute_login_async(config_snapshot=...)` **不传 cancel_event**。`execute_login_async` 第 186-187 行自动新建一个空 Event，第 192 行发现旧 future 未完成 → 命中去重 → 返回旧 future。手动登录 API 实际等待的是即将被取消的旧登录。

1. `engine.py` 传入新建的 cancel_event，并在超时后强制清理旧引用：


#### F10 — 定时任务 task_id 去重

**文件**: `app/services/task_executor.py:157-166`、`__init__`

**问题**: `execute_task_async` 直接 submit，无 task_id 去重。任务执行超过调度间隔时，下一分钟再次到期，提交第二个实例。

**修复**: 维护 task_id → future 映射，提交前检查是否已有 pending。

```python
# __init__ 新增
self._running_tasks: dict[str, Future] = {}
self._running_tasks_lock = threading.Lock()

def execute_task_async(self, task_id: str) -> Future:
    """异步执行定时任务（提交到 task_pool），带 task_id 去重。"""
    with self._running_tasks_lock:
        existing = self._running_tasks.get(task_id)
        if existing is not None and not existing.done():
            logger.info("定时任务 {} 已在执行中，跳过重复提交", task_id)
            return existing

        try:
            future = self._ensure_task_pool().submit(self.execute_task, task_id)
        except RuntimeError:
            # 队列已满
            raise
        self._running_tasks[task_id] = future

    # 锁外注册清理回调
    def _cleanup(f: Future, tid=task_id):
        with self._running_tasks_lock:
            if self._running_tasks.get(tid) is f:
                del self._running_tasks[tid]
    future.add_done_callback(_cleanup)
    return future
```

**注意**: `_execute_browser` 内部若调用 `execute_login_async`（不直接调用，但走 Worker CMD_LOGIN）不受此影响——浏览器任务的去重只针对同一 task_id。

#### F12 + F13 — 重构 _link_cancel_event，消除线程泄漏

**文件**: `app/services/task_executor.py:186-222`

**问题**: 每次去重命中都 `_link_cancel_event` → 新建 daemon 线程，最多存活 300s。高频手动登录时累积。

**修复**: 用单个常驻 watcher 线程 + 事件队列，替代每次 `start()`。F13（第 195 行冗余检查）在 F06 已一并处理。

```python
# __init__ 新增
self._cancel_link_queue: queue.Queue = queue.Queue()
self._cancel_link_thread: threading.Thread | None = None
self._cancel_link_lock = threading.Lock()

def _ensure_cancel_link_thread(self) -> None:
    """惰性启动单个 watcher 线程处理所有 cancel 联动。"""
    with self._cancel_link_lock:
        if self._cancel_link_thread is not None and self._cancel_link_thread.is_alive():
            return
        self._cancel_link_thread = threading.Thread(
            target=self._cancel_link_loop, daemon=True, name="cancel-link-watcher"
        )
        self._cancel_link_thread.start()

def _cancel_link_loop(self) -> None:
    """常驻 watcher：从队列取 (new_event, target_event) 并监控。"""
    pending: list[tuple[threading.Event, threading.Event]] = []
    while True:
        try:
            item = self._cancel_link_queue.get(timeout=1.0)
            if item is None:                # 毒丸，shutdown 时使用
                return
            pending.append(item)
        except queue.Empty:
            pass
        # 检查所有 pending 的新事件是否已 set
        still_pending = []
        for new_event, target_event in pending:
            if new_event.is_set():
                target_event.set()           # 联动，无需继续监控
            elif time.time() > getattr(new_event, "_deadline", 0):
                pass                         # 超时放弃（见下）
            else:
                still_pending.append((new_event, target_event))
        pending = still_pending

@staticmethod
def _link_cancel_event(new_event, target_event) -> None:
    """将联动请求入队，由常驻 watcher 处理（不再每次新建线程）。"""
    # 设 300s 截止时间
    new_event._deadline = time.time() + 300   # type: ignore[attr-defined]
    self._cancel_link_queue.put((new_event, target_event))
    self._ensure_cancel_link_thread()
```

**shutdown 配套**: `TaskExecutor.shutdown` 末尾投递毒丸：
```python

```

**注意**: `_link_cancel_event` 从 `@staticmethod` 改为实例方法（需要访问 `self._cancel_link_queue`）。调用处（execute_login_async 第 196 行）相应去掉类名前缀。`time` 模块需确认已 import（task_executor.py:14 已 import）。

#### F19 — 指数退避增加最大间隔上限

**文件**: `app/utils/retry.py:4-16`

**问题**: `exponential=True` 时间隔无限增长。当前硬编码 `exponential=False` 不触发，但作为通用工具缺乏防御。

**修复**:
```python
def get_retry_intervals(
    retry_interval: int,
    max_retries: int,
    *,
    exponential: bool = False,
    max_interval: int = 300,            # NEW: 上限 5 分钟
) -> list[int]:
    """计算重试间隔列表。

    exponential=True 时使用指数退避（间隔翻倍），否则使用固定间隔。
    指数退避受 max_interval 上限约束，避免间隔无限增长。
    """
    if exponential:
        return [min(retry_interval * (2**i), max_interval) for i in range(max_retries)]
    return [retry_interval] * max_retries
```
